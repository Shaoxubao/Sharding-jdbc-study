# 分库分表面试题及答案（商品价格系统实战版）

> 本文档基于 `sharding-jdbc-price` 商品价格分库分表系统实战，整理面试中高频出现的问题及参考答案，涵盖分片策略、数据迁移、双写一致性、Sharding-JDBC 原理等核心知识点。

---

## 一、分库分表基础

### 1. 什么时候需要分库分表？

**参考答案**：

分库分表是解决单库单表性能瓶颈的最终手段，通常在以下场景考虑：

1. **单表数据量过大**：MySQL 单表建议控制在 500 万~2000 万行以内（InnoDB 的 B+ 树三层结构极限约 2000 万行），超过后索引查询、全表扫描性能急剧下降。本系统 `product_price` 表已有 3000 万+ 数据，单表查询性能严重下降。
2. **单库连接数瓶颈**：应用并发高时，单个数据库的连接数（默认 151）容易被打满。
3. **单库磁盘/IO 瓶颈**：写操作集中导致磁盘 IO 饱和。
4. **数据隔离需求**：不同租户或业务线需要物理隔离。

> **注意**：分库分表是最后手段，在此之前应先尝试索引优化、SQL 优化、读写分离、缓存等手段。

---

### 2. 分库和分表的区别？先分库还是先分表？

**参考答案**：

| 维度 | 分库 | 分表 |
|------|------|------|
| 解决问题 | 连接数、磁盘 IO、CPU 瓶颈 | 单表数据量过大 |
| 物理位置 | 不同数据库实例/服务器 | 同一数据库内不同表 |
| 复杂度 | 高（跨库事务、Join 更难） | 相对较低 |
| 扩展性 | 扩展上限高 | 单库内扩展有限 |

**先分表还是先分库**：
- **先分表**：如果只是一张表数据量大，其他表正常，先分表（成本最低）。
- **再分库**：当单库连接数、磁盘、CPU 也撑不住时，再分库。
- **本系统方案**：同时进行分库+分表（4 库 × 8 表），因为 3000 万数据量下，仅分表到 32 张表（约 100 万/表），但单库仍承载 800 万数据和大量连接，需要同时分库分担连接和 IO 压力。

---

### 3. 分片策略有哪些？如何选择？

**参考答案**：

常见分片策略：

1. **取模分片（Hash）**：`product_id % 4`
   - 优点：数据分布均匀，避免热点。
   - 缺点：扩容时需要迁移大量数据（re-sharding）；范围查询效率低。
   - 适用：数据分布均匀、查询以等值为主。

2. **范围分片（Range）**：`0-1000万 -> db0, 1000万-2000万 -> db1`
   - 优点：扩容简单，只需增加新节点；范围查询高效。
   - 缺点：新数据集中在末尾节点，容易造成尾部热点。
   - 适用：时间序列数据、日志类数据。

3. **一致性哈希**：节点变化时只需迁移少量数据。
   - 优点：扩容时迁移数据量少。
   - 缺点：实现复杂，数据分布不如取模均匀。

4. **复合分片**：先按时间范围分库，再按 Hash 分表。

**本系统选择**：采用 `product_id % 4` 分库 + `product_id % 8` 分表（取模 Hash）。
- 原因是商品 ID 分布均匀，等值查询最多，数据不会倾斜。
- 价格更新频繁，Hash 分片能均匀分散写压力。

## 二、分片键与路由

### 4. 什么是分片键？如何选择？

**参考答案**：

**分片键（Sharding Column）** 是决定数据路由到哪个分片的字段。SQL 中若包含分片键条件，Sharding-JDBC 可直接定位到单个分片；否则需全分片扫描。

**选择原则**：
1. **高频查询条件**：最常见的 WHERE 条件字段。本系统按 `product_id` 查询是最核心场景。
2. **数据分布均匀**：避免某些值出现频率过高导致数据倾斜。
3. **不可变或极少变更**：分片键变更意味着数据迁移。
4. **避免跨分片查询**：如果经常需要按 `sku_id` 查询但分片键是 `product_id`，每次都会扫描全部分片。

**本系统选择 `product_id` 的原因**：
- 按商品 ID 查询是最常见的场景。
- 商品 ID 分布均匀，不会倾斜。
- 价格更新时天然携带 `product_id`。

---

### 5. 非分片键查询怎么办？（高频）

**参考答案**：

当查询条件不含分片键时，Sharding-JDBC 无法确定路由，会执行**全分片扫描**（广播到所有 32 张表），性能比单表更差。

**本系统中的问题**：
- `selectById(Long id)` —— `id` 不是分片键，全分片扫描。
- `selectBySkuId(Long skuId)` —— `sku_id` 不是分片键，全分片扫描。
- `count()` —— 全分片聚合。

**解决方案**：

1. **异构索引表（映射表）**
   - 新建 `sku_product_mapping` 表，以 `sku_id` 为分片键存储 `sku_id -> product_id` 映射。
   - 查询时先查映射表获取 `product_id`，再查主表。
   - 缺点是维护映射表的双写一致性。

2. **广播表（小表广播）**
   - 若 `sku_id` 到 `product_id` 的映射数据量可控（如 SKU 总数 100 万），可将映射表配置为广播表（每个分片都有完整副本）。
   - 查询映射表时无需路由，任意节点即可返回。

3. **覆盖索引 + 强制路由**
   - 在非分片键上建立索引，虽然仍需全分片扫描，但减少回表。
   - 对于 `id` 查询，由于 `id` 是主键，全分片扫描时每个分片走主键索引，效率尚可接受（但并发高时 32 次查询仍不可忽视）。

4. **更换分片键（业务妥协）**
   - 如果 `sku_id` 查询频率极高，考虑以 `sku_id` 作为分片键。但同一商品多个 SKU 会分散在不同分片，按 `product_id` 聚合查询又会变慢。

5. **ES/搜索引擎兜底**
   - 将数据同步到 Elasticsearch，非分片键的复杂查询走 ES，MySQL 只负责分片键精确查询。

### 6. 什么是数据倾斜？如何避免？

**参考答案**：

**数据倾斜**指分片后某些分片的数据量远大于其他分片，导致这些分片成为热点，失去分片的意义。

**常见原因**：
- 分片键选择不当，如按 `user_id` 分片但某些大 V 用户数据量远超普通用户。
- 按时间分片时新数据集中在最新分片（尾部热点）。

**避免方法**：
1. **Hash 取模**：若分片键本身分布均匀（如自增 ID、UUID），取模后数据自然均匀。本系统 `product_id` 分布均匀，Hash 分片不会倾斜。
2. **一致性哈希 + 虚拟节点**：减少节点变化时的不均匀。
3. **热点数据特殊处理**：大 V 用户单独分片或缓存。
4. **定期监控**：统计各分片数据量，发现倾斜及时调整。

## 三、Sharding-JDBC 原理

### 7. Sharding-JDBC 的执行流程是怎样的？

**参考答案**：

Sharding-JDBC 是** JDBC 驱动层**的分片中间件，对业务代码透明。执行流程如下：

1. **SQL 解析**：使用 ANTLR 解析 SQL 语法树，提取表名、列名、条件等。
2. **SQL 路由**：根据分片规则和 SQL 中的分片键值，计算目标数据源（DataNode）。
   - 若 SQL 含 `product_id = 5`，则路由到 `ds$->{5 % 4}.product_price_$->{5 % 8}`。
   - 若不含分片键，广播到所有数据节点。
3. **SQL 改写**：将逻辑表名改写为真实表名。
   - `SELECT * FROM product_price` → `SELECT * FROM product_price_0` ... `product_price_7`（共 32 条 SQL）。
4. **SQL 执行**：通过线程池并行执行路由后的真实 SQL。
5. **结果归并**：将多个分片的结果集合并（流式归并、内存归并、排序归并等）。
6. **返回结果**：封装为 ResultSet 返回给业务代码。

---

### 8. Sharding-JDBC 的 SQL 改写会做什么？

**参考答案**：

SQL 改写是将针对**逻辑表**的 SQL 转换为针对**真实物理表**的 SQL，主要包括：

1. **表名改写**：`product_price` → `product_price_2`
2. **分页改写**：LIMIT 10 OFFSET 20 需要改写为各分片的 LIMIT 并内存排序归并。
3. **聚合改写**：`COUNT(*)` 需要改写为各分片 `COUNT(*)` 后求和。
4. **ORDER BY 改写**：各分片分别排序后，Sharding-JDBC 进行流式归并排序。
5. **GROUP BY 改写**：各分片分组后，Sharding-JDBC 二次聚合。

**示例**：
```sql
-- 逻辑 SQL
SELECT * FROM product_price WHERE product_id = 5 ORDER BY id LIMIT 10;

-- 改写后（假设 product_id=5 路由到 ds1.product_price_5）
SELECT * FROM product_price_5 WHERE product_id = 5 ORDER BY id LIMIT 10;
```

### 9. Sharding-JDBC 支持的分布式主键生成策略有哪些？

**参考答案**：

Sharding-JDBC 内置多种分布式主键生成策略：

1. **SNOWFLAKE（雪花算法）**
   - 64 位 Long 类型，由 1 位符号 + 41 位时间戳 + 10 位机器 ID + 12 位序列号组成。
   - 优点：趋势递增，性能高，不依赖数据库。
   - 缺点：强依赖机器时钟，时钟回拨会产生重复 ID。
   - 本系统使用该策略，worker id 配置为不同值（如旧库 0，新库 1）。

2. **UUID**
   - 优点：全局唯一，生成简单。
   - 缺点：无序，字符串存储占用空间大，索引效率低。

3. **LEAF（美团开源）**
   - 基于号段模式（数据库预分配一批 ID），性能高。

4. **AUTO（数据库自增）**
   - 仅适用于单库单表，分片场景下不同分片会产生重复 ID。

**注意事项**：
- 分片场景下**禁止使用数据库 AUTO_INCREMENT**，不同分片会生成重复主键。
- 使用 SNOWFLAKE 时，insert SQL 必须显式包含 `id` 字段，由 ShardingSphere 在执行前自动生成并填充到实体对象中。

## 四、数据迁移方案

### 10. 不停服迁移数据的方案是什么？

**参考答案**：

本系统采用的**双写 + 全量迁移 + 增量同步 + 数据校验 + 流量切换**方案是业界标准做法。

**具体流程**：

1. **准备阶段**：
   - 创建新分片库和分片表（包括 `product_price` 和 `product_price_history`）。
   - 配置 Sharding-JDBC（统一 SNOWFLAKE 主键）。

2. **双写阶段**：
   - 启动双写开关，新写入的数据同时写入旧库和新库。
   - 旧库通过 `JdbcTemplate` 直接操作，新库通过 `ProductPriceMapper` 经 Sharding-JDBC 路由。
   - 读流量仍走旧库。

3. **全量数据迁移**：
   - 使用 `JdbcTemplate` 分页读取旧表数据。
   - 逐条插入新表（通过 Mapper 自动分片路由）。
   - 同时写入 `product_price_history` 表。

4. **增量数据同步**：
   - 基于 Binlog（Canal/Debezium）同步双写期间的数据变更。
   - **本系统代码中未实现**，生产环境必须接入。

5. **数据校验**：
   - 全量校验：逐条对比旧库和新库数据。
   - 抽样校验：随机 offset 分页查询获取真实记录进行对比。

6. **流量切换**：
   - 先切 10% 读流量到新库 → 50% → 100%。
   - 观察无误后停止双写，只写新库。

7. **清理阶段**：
   - 下线旧表，归档历史数据。

---

### 11. 双写方案中，旧库和新库如何保证一致性？

**参考答案**：

双写的核心挑战是**跨数据源事务**。Spring 的 `@Transactional` 只能保证单个数据源的事务，旧库和新库属于不同数据源，无法通过本地事务保证原子性。

**本系统的实现**：
- 旧库使用 `JdbcTemplate` 直接写入。
- 新库使用 `ProductPriceMapper` 经 Sharding-JDBC 写入。
- 两者在同一个方法中顺序执行，若新库写入失败，旧库数据已提交，造成不一致。

**解决方案**：

1. **ShardingSphere XA 分布式事务**
   ```java
   @ShardingTransactionType(TransactionType.XA)
   @Transactional
   public void insertWithDualWrite(...) { ... }
   ```
   - XA 协议保证强一致性，但性能较低，阻塞时间长。

2. **Seata AT 模式**
   - 基于 UNDO_LOG 实现最终一致性，性能较好。
   - 需要引入 Seata 服务端。

3. **异步对账补偿（本系统实际采用）**
   - 双写时允许短暂不一致。
   - 通过定时任务（`DataConsistencyChecker`）对比新旧库数据，发现不一致时补偿修复。
   - 优点：简单，性能高。
   - 缺点：存在短暂不一致窗口。

4. **消息队列保证最终一致性**
   - 写入旧库成功后发送 MQ 消息，消费者异步写入新库。
   - 配合死信队列和重试机制保证最终一致。

### 12. 数据校验时如何随机抽样？为什么不能直接随机生成 id？

**参考答案**：

**本系统的正确做法**：
```java
long randomOffset = (long) (random.nextDouble() * totalCount);
List<ProductPrice> samples = oldJdbcTemplate.query(
    "SELECT * FROM product_price ORDER BY id LIMIT 1 OFFSET ?",
    PRICE_ROW_MAPPER, (int) randomOffset);
```

**为什么不能直接随机生成 id**：
- 本系统使用 **SNOWFLAKE** 生成主键，不是连续递增的（含时间戳 + 机器位 + 序列号）。
- 随机生成的 Long 值大概率命中不存在的记录。
- 若 `totalCount > Integer.MAX_VALUE`，`(int) totalCount` 会溢出。

**正确做法**：
- 随机生成 `offset`（0 ~ totalCount-1）。
- 通过 `ORDER BY id LIMIT 1 OFFSET ?` 查询获取真实存在的记录。
- 再用该记录的 `id` 去新库查询对比。

## 五、历史表与关联设计

### 13. product_price_history 表的作用是什么？为什么也要分片？

**参考答案**：

**作用**：
1. **审计追踪**：记录每次价格新增/变更的快照，支持回溯价格变化历史。
2. **业务查询**：支持按商品查询历史价格列表。
3. **数据迁移辅助**：全量迁移旧数据时，将旧价格复制到 history 表。

**为什么也要分片**：
- 价格更新频繁，`history` 表的数据量增长可能比 `product_price` 更快（每次更新都新增一条记录）。
- 如果不分片，`history` 表会成为新的单表瓶颈。
- 本系统让 `history` 表与 `price` 表使用**完全相同的分片策略**（都按 `product_id` 分库分表），保证同一商品的价格和历史记录落在**同一个数据库和同一张物理表**。
- 这样查询历史价格时，Sharding-JDBC 可以根据 `product_id` 直接路由到单个分片，无需跨分片关联。

---

### 14. 如果历史表数据量太大，如何进一步优化？

**参考答案**：

1. **冷热数据分离**
   - 近 90 天的热数据保留在 MySQL 分片表中。
   - 90 天前的冷数据通过定时任务迁移到 TiDB/ClickHouse/HBase 等存储。

2. **归档表**
   - 按月创建归档表（`product_price_history_202401`、`product_price_history_202402`）。
   - 但注意分片键是 `product_id`，按时间归档需要跨分片扫描，效率较低。可考虑异构存储（如将历史数据同步到 ES 按时间范围查询）。

3. **TTL 自动过期**
   - MySQL 8.0 支持行级 TTL，自动删除超期数据。

4. **精简字段**
   - history 表只保留变更的字段和变更时间，减少单条记录大小。

## 六、性能与优化

### 15. 分库分表后，COUNT(*) 和分页查询为什么会变慢？

**参考答案**：

**COUNT(*) 变慢**：
- 逻辑 SQL `SELECT COUNT(*) FROM product_price` 会被改写为 32 条 SQL（4 库 × 8 表）。
- Sharding-JDBC 并行执行 32 条 SQL，然后将结果求和归并。
- 即使并行，32 次查询的网络 RTT 和数据库 CPU 消耗也不可忽视。
- **优化**：在每个分片维护计数器表，或引入 Redis 计数。

**分页查询变慢**：
- `LIMIT 10 OFFSET 100000` 需要改写为各分片的 LIMIT 并内存排序。
- 若 ORDER BY 非分片键，Sharding-JDBC 需要从各分片取回大量数据内存归并。
- **优化**：避免深分页；使用游标分页（`WHERE id > ? LIMIT 10`）；或分页查询时强制带分片键。

---

### 16. 如何评估分库分表后的性能提升？

**参考答案**：

可以从以下维度评估：

1. **查询性能**
   - 分片键等值查询：从全表扫描变为单分片索引查询，性能提升数十倍。
   - 非分片键查询：可能更慢（全分片扫描），需单独优化。

2. **写入性能**
   - 写操作分散到多个库，磁盘 IO 和连接数压力线性下降。
   - 但事务和一致性开销增加。

3. **并发能力**
   - 4 个数据库实例理论上支持 4 倍的连接数和吞吐量。

4. **本系统预期效果**
   - 按 `product_id` 查询性能提升 80%+。
   - 写入吞吐量提升 3~4 倍。
   - 数据分布均匀，单表控制在 100 万行以内。

## 七、实战代码相关问题

### 17. 本系统代码中，为什么 ProductPriceMigrationService 要用 JdbcTemplate 操作旧库，而不是直接用 Mapper？

**参考答案**：

在 `application-migration.yml` 中，`old-ds` 和 `ds0-ds3` 都被注册到 Sharding-JDBC 的统一数据源中。

若使用 `ProductPriceMapper` 操作旧库：
- Mapper 的 SQL 会被 Sharding-JDBC 拦截解析。
- 由于 `product_price` 配置了分片规则，`Sharding-JDBC` 会将所有对 `product_price` 的 SQL 路由到 `ds0-ds3` 的分片表。
- 即使意图是写入 `old-ds.product_price`，Sharding-JDBC 也会错误地路由到新库。

**解决方案**：
- 通过 `@Qualifier("old-ds")` 注入原始的 `DataSource`。
- 使用 `JdbcTemplate` 直接在该数据源上执行 SQL，**完全绕过 Sharding-JDBC 的 SQL 解析和路由**。
- 这样旧库操作和新库操作互不干扰。

### 18. 本系统中 update SQL 为什么要加 product_id 条件？

**参考答案**：

原始代码：
```sql
UPDATE product_price SET ... WHERE id = #{id}
```

由于 `id` 不是分片键，Sharding-JDBC 无法确定该 `id` 属于哪个分片，只能将 SQL 广播到**全部 32 张表**执行。

修复后：
```sql
UPDATE product_price SET ... WHERE id = #{id} AND product_id = #{productId}
```

Sharding-JDBC 解析 WHERE 条件时，发现包含分片键 `product_id`，可以直接计算路由到单个分片，避免了全分片扫描。

> **注意**：业务代码调用 update 时必须传入 `productId`，这要求接口设计和业务逻辑配合。

### 19. 如果生产环境需要从 4 库 × 8 表扩容到 8 库 × 8 表，如何操作？

**参考答案**：

**取模分片的扩容难点**：
- 原规则 `product_id % 4`，扩容到 `product_id % 8` 后，约 75% 的数据需要迁移（只有 `product_id % 8 == product_id % 4` 的数据不需要迁）。

**扩容方案**：

1. **翻倍扩容法（推荐）**
   - 4 库 × 8 表 → 8 库 × 8 表。
   - 每个旧库的数据按 `product_id % 8` 拆分到两个新库。
   - 只需迁移 50% 的数据，而非全量。

2. **一致性哈希**
   - 使用一致性哈希环，新增节点时只影响相邻节点的数据，迁移量最小。

3. **双写 + 数据迁移（类似本系统的迁移方案）**
   - 创建新分片库。
   - 启动双写（同时写入新旧分片）。
   - 全量迁移历史数据。
   - 切读流量到新分片。

4. **使用 ShardingSphere 的自动分片（未来方向）**
   - ShardingSphere 5.x 支持自动分片（AutoTable），可根据数据量自动创建新分片。

### 20. 分库分表后，JOIN 查询和事务如何处理？

**参考答案**：

**JOIN 查询**：
- **同库同表 JOIN**：完全支持，性能不受影响。
- **同库不同表 JOIN**：若两张表在同一数据库（如都在 `ds0`），MySQL 原生支持。
- **跨库 JOIN**：Sharding-JDBC 不支持跨库 JOIN，需要在应用层拆分为多次查询，或数据冗余。
- **本系统**：`product_price` 和 `product_price_history` 使用相同分片策略，同一商品的记录保证在同库同表，因此按 `product_id` 关联时实际上是同库 JOIN，性能良好。

**事务**：
- **同库事务**：`@Transactional` 完全支持。
- **跨库事务**：本地事务不支持，需使用：
  - ShardingSphere XA 事务（强一致，性能低）。
  - Seata AT 模式（最终一致，性能较好）。
  - TCC/Saga 模式（业务侵入大）。
- **本系统**：双写属于跨库操作，当前未使用分布式事务，依赖异步对账补偿保证最终一致性。

## 八、扩展问题

### 21. ShardingSphere-JDBC 和 ShardingSphere-Proxy 的区别？

**参考答案**：

| 维度 | ShardingSphere-JDBC | ShardingSphere-Proxy |
|------|---------------------|----------------------|
| 架构位置 | JDBC 驱动层（客户端） | 数据库代理层（服务端） |
| 接入方式 | 引入 Jar 包，代码配置 | 独立部署，像数据库一样连接 |
| 性能 | 无网络转发，性能更高 | 多一次网络转发，略低 |
| 语言兼容 | 仅 Java | 任意语言（MySQL/PostgreSQL 协议） |
| 运维成本 | 低（随应用部署） | 高（需独立运维代理集群） |
| 连接数 | 占用应用连接池 | 代理层统一连接，后端连接可控 |

**选型建议**：
- Java 应用、追求性能、团队熟悉 Java → JDBC。
- 多语言环境、希望统一管控、连接数紧张 → Proxy。

### 22. 分库分表后，如何平滑下线旧系统？

**参考答案**：

1. **读流量灰度切换**
   - 先切换 1% → 10% → 50% → 100% 读流量到新库。
   - 每个阶段观察错误率、延迟、数据一致性。

2. **写流量切换**
   - 停止双写中的旧库写入。
   - 只保留新库写入。

3. **保留旧库只读**
   - 旧库改为只读模式，保留一段时间作为应急回滚。

4. **数据归档**
   - 将旧库数据备份到冷存储（OSS、HDFS）。

5. **完全下线**
   - 确认无问题后，下线旧库实例，释放资源。
