# 商品价格表分表实战实现细节文档

## 一、项目概述

### 1.1 场景描述
某电商平台商品价格系统，商品价格表（product_price）已有 3000 万+ 数据，单表查询性能下降严重，需要使用 Sharding-JDBC 进行分表优化。

### 1.2 核心需求
1. 价格系统不能停服，需要在线迁移
2. 查询场景：按商品 ID 查询价格、按 SKU ID 查询价格、按价格范围查询
3. 写入场景：价格更新频繁，需要保证数据一致性
4. 历史数据查询：需要查询历史价格记录

### 1.3 技术选型
- **分片框架**：Sharding-JDBC 4.0.0-RC1
- **数据库**：MySQL 5.7+
- **ORM框架**：MyBatis
- **连接池**：Druid

### 1.4 已知限制（重要）
本项目是一个学习性质的 Demo，以下能力在代码中有明确实现，但生产环境需要进一步完善：
1. **跨数据源事务**：当前使用 Spring `@Transactional` 只能保证单个数据源的事务，双写时若旧库和新库操作需要原子性，需引入 ShardingSphere 分布式事务（XA/Seata）
2. **非分片键查询**：按 `id`、`sku_id` 查询会触发 Sharding-JDBC 全分片扫描，生产环境建议添加广播表或异构索引表
3. **Binlog 增量同步**：文档方案提及，但代码中未实现，生产环境需接入 Canal/Debezium

## 二、分表方案设计

### 2.1 分片策略

采用**按商品 ID 分片**的策略，原因如下：
1. 按商品 ID 查询是最常见的场景，性能最优
2. 数据分布均匀，避免数据倾斜
3. 配置简单，维护成本低
4. 适合价格更新频繁的场景

### 2.2 分片规则

**分库规则**：4个数据库，按 product_id % 4 分库
**分表规则**：每个数据库8张表，按 product_id % 8 分表

**数据节点**：
- 价格表：`ds$->{0..3}.product_price_$->{0..7}`
- 历史表：`ds$->{0..3}.product_price_history_$->{0..7}`

共 4 × 8 = 32 张物理表（价格表 + 历史表各 32 张）

### 2.3 分片算法

使用 Inline 表达式实现分片算法：

```yaml
# 分库算法
database-strategy:
  inline:
    sharding-column: product_id
    algorithm-expression: ds$->{product_id % 4}

# 分表算法
table-strategy:
  inline:
    sharding-column: product_id
    algorithm-expression: product_price_$->{product_id % 8}
```

历史表使用**与价格表完全一致**的分片策略，保证同一商品的价格记录和历史记录落在**同一个数据库和同一张物理表**，便于本地关联查询。

### 2.4 主键生成策略

所有表（包括旧表）统一使用 **SNOWFLAKE** 算法生成分布式主键，避免不同数据源主键冲突：

```yaml
key-generator:
  column: id
  type: SNOWFLAKE
  props:
    worker:
      id: 1  # 不同实例需配置不同 worker id
```

> **注意**：所有 insert SQL 必须显式包含 `id` 字段，由 ShardingSphere 在 SQL 执行前自动生成并填充到实体中。

## 三、数据库设计

### 3.1 表结构

**商品价格表（product_price）**

```sql
CREATE TABLE product_price (
    id BIGINT PRIMARY KEY,  -- SNOWFLAKE 生成，非自增
    product_id BIGINT NOT NULL COMMENT '商品ID',
    sku_id BIGINT NOT NULL COMMENT 'SKU ID',
    price DECIMAL(10,2) NOT NULL COMMENT '价格',
    original_price DECIMAL(10,2) COMMENT '原价',
    cost_price DECIMAL(10,2) COMMENT '成本价',
    start_time DATETIME COMMENT '生效开始时间',
    end_time DATETIME COMMENT '生效结束时间',
    status TINYINT DEFAULT 1 COMMENT '状态：1-有效，0-无效',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_product_id (product_id),
    INDEX idx_sku_id (sku_id),
    INDEX idx_price (price),
    INDEX idx_create_time (create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='商品价格表';
```

**商品价格历史表（product_price_history）**

```sql
CREATE TABLE product_price_history (
    id BIGINT PRIMARY KEY,  -- SNOWFLAKE 生成，非自增
    price_id BIGINT NOT NULL COMMENT '原价格表ID',
    product_id BIGINT NOT NULL COMMENT '商品ID',
    sku_id BIGINT NOT NULL COMMENT 'SKU ID',
    price DECIMAL(10,2) NOT NULL COMMENT '价格',
    original_price DECIMAL(10,2) COMMENT '原价',
    cost_price DECIMAL(10,2) COMMENT '成本价',
    start_time DATETIME COMMENT '生效开始时间',
    end_time DATETIME COMMENT '生效结束时间',
    status TINYINT DEFAULT 1 COMMENT '状态',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_product_id (product_id),
    INDEX idx_sku_id (sku_id),
    INDEX idx_price_id (price_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='商品价格历史表';
```

### 3.2 数据库分布

**旧数据库**：
- price_db：包含单表 product_price、product_price_history

**新数据库（分片）**：
- price_db_0：product_price_0 ~ product_price_7，product_price_history_0 ~ product_price_history_7
- price_db_1：同上
- price_db_2：同上
- price_db_3：同上

### 4.2 Mapper 接口

**ProductPriceMapper.java**

提供以下方法：
- `insert(ProductPrice productPrice)` - 插入价格（**id 由 SNOWFLAKE 预生成，SQL 显式包含 id 字段**）
- `update(ProductPrice productPrice)` - 更新价格（**WHERE 条件包含 product_id 分片键，避免全分片扫描**）
- `selectById(Long id)` - 根据ID查询（⚠️ 非分片键，触发全分片扫描）
- `selectByProductId(Long productId)` - 根据商品ID查询当前价格（命中分片键，性能最优）
- `selectBySkuId(Long skuId)` - 根据SKU ID查询当前价格（⚠️ 非分片键，触发全分片扫描）
- `selectByPriceRange(...)` - 按价格范围查询（包含 product_id 分片键，单分片查询）
- `selectHistoryByProductId(...)` - 查询当前价格表中某商品按时间排序的记录
- `count()` - 统计总数（⚠️ 触发全分片聚合）
- `selectByPage(...)` - 分页查询（⚠️ 触发全分片排序聚合）
- `deleteById(Long id)` - 删除价格（⚠️ 非分片键，触发全分片扫描）

**ProductPriceHistoryMapper.java**

提供以下方法：
- `insert(ProductPriceHistory history)` - 插入历史记录
- `selectByProductId(Long productId, int limit)` - 根据商品ID查询历史价格

### 4.3 Service 层

**ProductPriceService.java**

提供基础的价格管理功能：
- `addPrice(ProductPrice productPrice)` - 添加价格，同时写入 history 表
- `updatePrice(ProductPrice productPrice)` - 更新价格，将**旧值**写入 history 表
- `getCurrentPrice(Long productId)` - 查询当前价格（命中分片键）
- `getSkuPrice(Long skuId)` - 查询SKU价格（非分片键，全分片扫描）
- `queryByPriceRange(...)` - 按价格范围查询（含分片键，单分片）
- `queryHistoryPrice(...)` - 查询历史价格（查询 **product_price_history** 表）

**ProductPriceMigrationService.java**

实现双写逻辑（迁移期间使用）：
- `insertWithDualWrite(ProductPrice productPrice)` - 双写插入
  - 使用 **JdbcTemplate** 直接操作旧库 `old-ds`
  - 使用 `ProductPriceMapper` 经 Sharding-JDBC 路由写入新库分片表
  - 写入 history 表
- `updateWithDualWrite(ProductPrice productPrice)` - 双写更新
  - 先从旧库查询旧值，保存到 history 表
  - 使用 JdbcTemplate 更新旧库
  - 使用 Mapper 更新新库

> **为什么使用 JdbcTemplate 操作旧库？**
> 在 migration profile 下，Sharding-JDBC 将 `old-ds` 和 `ds0-ds3` 统一管理。若使用同一个 `ProductPriceMapper`，Sharding-JDBC 会根据 `product_price` 的分片规则将所有 SQL 路由到 `ds0-ds3`，**无法直接写入 `old-ds`**。因此通过 `@Qualifier("old-ds")` 注入原始数据源，使用 `JdbcTemplate` 直接执行 SQL，绕过 Sharding-JDBC 路由。

**ProductPriceReadService.java**

实现读写切换和双读对比：
- `selectByProductId(Long productId)` - 根据配置选择读源（old/new/both）
- `selectById(Long id)` - 根据配置选择读源
- **both 模式**：同时读取旧库（JdbcTemplate）和新库（Mapper），对比数据一致性

**ProductPriceMigrationJob.java**

实现数据迁移功能：
- `migrateAllData()` - 全量数据迁移（JdbcTemplate 分页读取旧库 -> Mapper 批量写入新库）
- `migrateDataConcurrently()` - 并发数据迁移（线程池 + JdbcTemplate）

**DataConsistencyChecker.java**

实现数据校验功能：
- `checkAllData()` - 全量数据校验（JdbcTemplate 分页读取旧库 -> Mapper 逐条查询新库对比）
- `randomCheck()` - 抽样校验（**随机 offset 分页查询获取真实存在的记录**，避免随机生成不存在的 id）

## 六、迁移方案

### 6.1 迁移流程

**阶段一：准备阶段（1-2天）**
1. 创建新数据库和分片表（包括 price 和 history）
2. 配置 Sharding-JDBC（统一 SNOWFLAKE 主键）
3. 测试环境验证

**阶段二：双写阶段（3-5天）**
1. 实现双写逻辑（JdbcTemplate 写旧库 + Mapper 写新库）
2. 配置双写开关
3. 实现读写分离逻辑（JdbcTemplate 读旧库 + Mapper 读新库）
4. 灰度发布双写功能

**阶段三：历史数据迁移（5-7天）**
1. 全量数据迁移（JdbcTemplate 分页读旧 -> Mapper 批量写新）
2. 增量数据同步（基于 Binlog，**生产环境需接入 Canal/Debezium**，代码中未实现）
3. 迁移进度监控

**阶段四：数据校验（2-3天）**
1. 全量数据校验
2. 抽样校验（随机 offset 分页查询，避免命中不存在的 id）
3. 数据一致性检查

**阶段五：流量切换（1-2天）**
1. 切换 10% 读流量到新表
2. 切换 50% 读流量到新表
3. 切换 100% 读流量到新表
4. 停止双写，只写新表

**阶段六：清理阶段（1天）**
1. 下线旧表
2. 清理旧数据
3. 归档历史数据

### 6.2 双写实现

```java
@Transactional
public void insertWithDualWrite(ProductPrice productPrice) {
    productPrice.setCreateTime(new Date());
    productPrice.setUpdateTime(new Date());
    productPrice.setStatus(1);
    
    // 使用 JdbcTemplate 直接写入旧库，绕过 Sharding-JDBC 路由
    if (dualWriteEnabled && oldTableWriteEnabled) {
        oldJdbcTemplate.update(
            "INSERT INTO product_price (id, product_id, ...) VALUES (?, ?, ...)",
            productPrice.getId(), productPrice.getProductId(), ...
        );
    }
    
    // 使用 Mapper 经 Sharding-JDBC 路由写入新库分片表
    if (dualWriteEnabled && newTableWriteEnabled) {
        newPriceMapper.insert(productPrice);
    }
    
    ProductPriceHistory history = new ProductPriceHistory();
    BeanUtils.copyProperties(productPrice, history);
    history.setPriceId(productPrice.getId());
    historyMapper.insert(history);
}
```

**跨数据源事务说明**：
当前代码使用 Spring `@Transactional`，只能保证**单个数据源**的事务。若旧库写入成功但新库写入失败，数据会不一致。生产环境双写应使用：
- ShardingSphere XA 事务（`@ShardingTransactionType(TransactionType.XA)`）
- 或 Seata AT 模式

### 6.3 数据迁移实现

**全量迁移**：
- 使用 JdbcTemplate 分页查询旧表数据
- 使用 Mapper 批量插入新表
- 记录失败数据，支持重试

**并发迁移**：
- 使用线程池并发迁移
- 使用 JdbcTemplate 保证每个线程独立读取旧库
- 控制并发度，避免数据库压力过大

### 6.4 数据校验实现

**全量校验**：
- JdbcTemplate 逐页读取旧表数据
- Mapper 逐条查询新表数据
- 记录不一致数据，生成校验报告

**抽样校验**：
- 随机生成 offset，使用 JdbcTemplate 分页查询获取**真实存在的记录**
- 对比数据一致性
- 快速验证迁移质量

> **为什么不直接随机生成 id？**
> 因为 SNOWFLAKE 生成的 id 不是连续递增的，随机数大概率命中不存在的记录。通过随机 offset 分页查询可以确保获取真实数据。

## 七、关键问题与优化建议

### 7.1 非分片键查询风险

当前分片键为 `product_id`，以下查询会触发**全分片扫描**，性能比单表更差：

| 查询条件 | 影响 |
|---------|------|
| `WHERE id = ?` | 全 32 张表扫描 |
| `WHERE sku_id = ?` | 全 32 张表扫描 |
| `COUNT(*)` | 全 32 张表聚合 |
| `ORDER BY id LIMIT` | 全 32 张表排序聚合 |

**优化方案**：
1. **异构索引表**：为 `sku_id` 创建独立的索引映射表（sku_id -> product_id），先查映射表获取 product_id，再查主表
2. **广播表**：若 `sku_id` 到 `product_id` 的映射数据量不大，可将映射表配置为广播表（每个分片都有完整副本）
3. **覆盖索引**：为高频查询的非分片键创建覆盖索引，减少回表开销（但无法避免全分片扫描）

### 7.2 跨数据源事务

双写时旧库和新库属于不同数据源，Spring 本地事务无法保证原子性。

**优化方案**：
1. 使用 ShardingSphere XA 分布式事务
2. 使用 Seata AT 模式
3. 或者采用"异步对账补偿"机制：允许短暂不一致，通过定时任务对比并修复

### 7.3 product_price_history 表设计

`product_price_history` 与 `product_price` 使用**相同的分片策略**，优势：
- 同一商品的价格和历史记录在同一库表，关联查询性能好
- 避免历史表成为新的单表瓶颈

注意事项：
- history 表数据量增长快，可配置**历史数据归档策略**（如 90 天前的数据迁移到冷存储）
- 归档时按 `create_time` 范围操作，注意分片键是 `product_id`，跨分片的批量删除效率较低

## 九、总结
