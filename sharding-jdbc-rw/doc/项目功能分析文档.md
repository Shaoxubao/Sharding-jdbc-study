# Sharding-JDBC 读写分离与分库分表项目详细分析

## 项目概述

本项目是一个基于 Sharding-JDBC 4.0.0-RC1 版本的数据库中间件示例项目，实现了多种数据库分片策略和读写分离功能。项目使用 Spring Boot 2.3.7 作为基础框架，集成 MyBatis 进行数据持久化操作。

## 技术栈

### 核心框架
- **Spring Boot**: 2.3.7.RELEASE
- **Sharding-JDBC**: 4.0.0-RC1
- **MyBatis**: 2.1.4
- **MySQL**: 使用 MySQL Connector/J
- **Druid**: 1.1.21（阿里巴巴数据库连接池）
- **Lombok**: 用于简化实体类开发

### 开发环境
- **Java**: 1.8
- **构建工具**: Maven
- **服务器端口**: 8086

## 核心功能实现

### 1. 读写分离功能

#### 配置文件：application-rw.yml

#### 数据源配置
项目配置了三个数据源：
- **ds1**: 主库（Master），负责写操作
  - 地址：192.168.170.128:3306
  - 数据库：sharding-jdbc-demo
  
- **ds2**: 从库（Slave 1），负责读操作
  - 地址：192.168.170.129:3306
  
- **ds3**: 从库（Slave 2），负责读操作
  - 地址：192.168.170.129:3306

#### 读写分离配置
```yaml
masterslave:
  name: ms
  master-data-source-name: ds1        # 主库
  slave-data-source-names: ds2,ds3   # 从库列表
  load-balance-algorithm-type: round_robin  # 轮询负载均衡
```

#### 实现原理
1. **写操作路由**：所有 INSERT、UPDATE、DELETE 操作自动路由到主库 ds1
2. **读操作路由**：SELECT 操作根据轮询算法在 ds2 和 ds3 之间分配
3. **负载均衡**：采用 round_robin（轮询）策略，确保读请求均匀分布到各个从库
4. **默认数据源**：配置 default-data-source-name: ds1 确保写操作正确路由

#### 关键配置说明
- `sql.show: true`：显示 SQL 执行日志，便于调试
- `default-data-source-name: ds1`：必须配置，否则所有节点会被视为从库，导致写操作失败

### 2. 标准分片策略

#### 配置文件：application-standard.yml

#### 分片配置
```yaml
tables:
  ksd_user:
    key-generator:
      column: id
      type: SNOWFLAKE              # 雪花算法生成分布式ID
    actual-data-nodes: ds$->{0..1}.ksd_user$->{0..1}  # 2库2表
    database-strategy:
      standard:
        sharding-column: birthday   # 按生日分库
        preciseAlgorithmClassName: com.baoge.algorithm.BirthdayAlgorithm
    table-strategy:
      inline:
        sharding-column: age        # 按年龄分表
        algorithm-expression: ksd_user$->{age % 2}
```

#### 数据节点分布
- **数据库**：ds0, ds1（2个数据库）
- **表**：每个数据库包含 ksd_user0, ksd_user1（2张表）
- **总节点数**：4个物理表（ds0.ksd_user0, ds0.ksd_user1, ds1.ksd_user0, ds1.ksd_user1）

#### 分片算法

##### 1. 生日分库算法（BirthdayAlgorithm.java）
```java
public class BirthdayAlgorithm implements PreciseShardingAlgorithm<Date> {
    List<Date> dateList = new ArrayList<>();
    {
        Calendar calendar2 = Calendar.getInstance();
        calendar2.set(2021, 1, 1, 0, 0, 0);
        Calendar calendar3 = Calendar.getInstance();
        calendar3.set(2023, 1, 1, 0, 0, 0);
        dateList.add(calendar2.getTime());
        dateList.add(calendar3.getTime());
    }

    @Override
    public String doSharding(Collection<String> collection, PreciseShardingValue<Date> preciseShardingValue) {
        Date date = preciseShardingValue.getValue();
        Iterator<String> iterator = collection.iterator();
        String target = null;
        for (Date s : dateList) {
            target = iterator.next();
            if (date.before(s)) {
                break;
            }
        }
        return target;
    }
}
```

**算法逻辑**：
- 定义两个时间节点：2021-02-01 和 2023-02-01
- 用户生日早于 2021-02-01：路由到 ds0
- 用户生日在 2021-02-01 到 2023-02-01 之间：路由到 ds1
- 用户生日晚于 2023-02-01：路由到 ds1（最后一个）

##### 2. 年龄分表算法（Inline 表达式）
```yaml
algorithm-expression: ksd_user$->{age % 2}
```

**算法逻辑**：
- 年龄为偶数：路由到 ksd_user0
- 年龄为奇数：路由到 ksd_user1

#### 主键生成策略
- **算法**：SNOWFLAKE（雪花算法）
- **特点**：分布式环境下生成唯一ID，保证全局唯一性和有序性
- **配置**：数据库主键不能使用自增长

### 4. 年月分片策略

#### 配置文件：application-yearmonth.yml

#### 分片配置
```yaml
tables:
  ksd_user:
    key-generator:
      column: id
      type: SNOWFLAKE
    actual-data-nodes: ds$->{0..1}.ksd_user$->{0..1}
    database-strategy:
      standard:
        sharding-column: birthday
        preciseAlgorithmClassName: com.baoge.algorithm.BirthdayAlgorithm
    table-strategy:
      inline:
        sharding-column: age
        algorithm-expression: ksd_user$->{age % 2}

  ksd_order:
    actual-data-nodes: ds0.ksd_order$->{0..1}
    key-generator:
      column: order_id
      type: SNOWFLAKE
    table-strategy:
      inline:
        sharding-column: orderid
        algorithm-expression: ksd_order$->{orderid % 2}

  ksd_user_order:
    actual-data-nodes: ds0.ksd_user_order_$->{2021..2022}${(1..3).collect{t ->t.toString().padLeft(2,'0')} }
    key-generator:
      column: order_id
      type: SNOWFLAKE
    database-strategy:
      inline:
        sharding-column: order_id
        algorithm-expression: ds$->{order_id % 2}
    table-strategy:
      standard:
        shardingColumn: yearmonth
        preciseAlgorithmClassName: com.baoge.algorithm.YearMonthShardingAlgorithm
```

#### 表结构说明

##### 1. ksd_user 表
- 与标准策略相同，按生日分库，按年龄分表

##### 2. ksd_order 表
- **数据节点**：ds0.ksd_order0, ds0.ksd_order1（单库2表）
- **分片键**：orderid
- **分片算法**：按 orderid 取模分表

##### 3. ksd_user_order 表（按年月分表）
- **数据节点**：ds0.ksd_user_order_202101 到 ds0.ksd_user_order_202212
- **表数量**：2年 × 12月 = 24张表
- **分片键**：yearmonth（格式：YYYYMM）
- **分片算法**：使用 YearMonthShardingAlgorithm

#### 年月分片算法实现

```java
public class YearMonthShardingAlgorithm implements PreciseShardingAlgorithm<String> {
    private static final String SPLITTER = "_";

    @Override
    public String doSharding(Collection availableTargetNames, PreciseShardingValue shardingValue) {
        String tbName = shardingValue.getLogicTableName() + "_" + shardingValue.getValue();
        System.out.println("Sharding input:" + shardingValue.getValue() + ", output:{}" + tbName);
        return tbName;
    }
}
```

**算法逻辑**：
- 输入：年月字符串（如 "202203"）
- 输出：表名（如 "ksd_user_order_202203"）
- 直接将年月值拼接到逻辑表名后，形成物理表名

**数据节点生成表达式解析**：
```groovy
ds0.ksd_user_order_$->{2021..2022}${(1..3).collect{t ->t.toString().padLeft(2,'0')} }
```
- `2021..2022`：生成 2021 和 2022 两个年份
- `(1..3).collect{t ->t.toString().padLeft(2,'0')}`：生成 01, 02, 03（示例中只配置了前3个月）
- 实际应该扩展为 1..12 生成完整的12个月

@Mapper
public interface UserMapper {
    @Insert("insert into ksd_user(nickname,password,sex,birthday,age) values(#{nickname},#{password},#{sex},#{birthday},#{age})")

## 分片策略总结

### 1. 读写分离（rw）
- **适用场景**：读多写少的业务场景
- **优势**：提高读性能，减轻主库压力
- **配置**：1主2从，轮询负载均衡

### 2. 标准分片（standard）
- **适用场景**：需要自定义分片逻辑的复杂场景
- **分片键**：生日（分库）、年龄（分表）
- **算法**：PreciseShardingAlgorithm + Inline 表达式

### 3. 子表分片（sub）
- **适用场景**：简单的取模分片
- **分片键**：性别（分库）、年龄（分表）
- **算法**：Inline 表达式

### 4. 年月分片（yearmonth）
- **适用场景**：按时间维度分表，如订单表、日志表
- **分片键**：年月（yearmonth）
- **算法**：自定义 PreciseShardingAlgorithm
- **特点**：支持多表多维度分片

## 核心技术点

### 1. Sharding-JDBC 分片算法类型

#### PreciseShardingAlgorithm（精确分片算法）
- 用于处理 = 和 IN 的分片
- 实现类：BirthdayAlgorithm、YearMonthShardingAlgorithm
- 特点：返回单个目标数据源或表

#### Inline（行表达式分片算法）
- 使用 Groovy 表达式语法
- 适合简单的取模、范围分片
- 示例：`ds$->{sex % 2}`、`ksd_user$->{age % 2}`

### 2. 主键生成策略

#### SNOWFLAKE（雪花算法）
- **特点**：
  - 分布式环境下生成唯一ID
  - 时间戳 + 机器ID + 序列号
  - 趋势递增，有利于索引性能
- **优势**：
  - 无需数据库自增
  - 避免ID冲突
  - 性能高效

### 3. 数据源配置

#### Druid 连接池配置
```yaml
type: com.alibaba.druid.pool.DruidDataSource
driver-class-name: com.mysql.cj.jdbc.Driver
url: jdbc:mysql://...
username: root
password: 123456
maxPoolSize: 100
minPoolSize: 5
```

**参数说明**：
- `maxPoolSize`：最大连接数
- `minPoolSize`：最小连接数
- 使用 Druid 监控功能（需额外配置）

### 4. 分片规则配置

#### actual-data-nodes（数据节点）
- 格式：`数据源$->{范围}.逻辑表名$->{范围}`
- 示例：`ds$->{0..1}.ksd_user$->{0..1}`
- 作用：定义物理表的实际分布

#### sharding-column（分片键）
- 选择原则：
  - 查询频繁的字段
  - 数据分布均匀的字段
  - 避免使用频繁更新的字段

#### 分片策略
- **database-strategy**：分库策略
- **table-strategy**：分表策略
- 两者可以独立配置

## 注意事项

### 1. 主从复制
- 读写分离功能依赖 MySQL 主从复制
- 需要提前配置主从同步

### 2. 表结构创建
- 物理表需要手动创建
- 表结构与逻辑表一致
- 分片表需要创建所有节点

### 3. 分片键选择
- 分片键一旦确定，不易修改
- 建议选择查询频繁且数据分布均匀的字段

### 4. 跨分片查询
- Sharding-JDBC 对跨分片查询支持有限
- 避免使用 JOIN、GROUP BY 等复杂查询

### 5. 事务支持
- 分片事务较为复杂
- 建议使用柔性事务或最终一致性
